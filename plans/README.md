## Conventions every plan enforces

All plans target Kntnt's coding standard (Plan 001 installs it into `agents.d/coding-standard/`): `declare(strict_types=1)`; WordPress flavour formatting (tabs, padded parens, `$snake_case`, `Pascal_Snake_Case` classes); `[ ... ]` arrays with trailing commas; one class per file, filename = class name under `classes/`; PHPDoc with `@since 1.0.0`; English source strings wrapped in `__()`/`esc_html__()` with text domain `kntnt-autolink`; escape at output, sanitise every superglobal, capability + nonce on every admin action; the engine (`Linker`, `Ruleset`, `Keyword`) stays free of WordPress calls so it unit-tests without a bootstrap. Modern PHP 8.4.

Verification gates (defined in Plan 001): `composer test` (Pest unit suite), `composer analyse` (PHPStan level max + WordPress stubs), `php -l` (syntax), and `bash tests/Integration/run.sh` (Playground, Plan 008).

## Notes for the executor models

- These plans were written against the design doc, not against existing code (there is none yet). Each plan's "Current state" describes what the *previous* plans will have produced; if reality differs, the drift checks tell you to STOP rather than improvise.
- The API contracts (class/method signatures) are repeated across plans deliberately so each is self-contained. They must stay consistent: if you change a signature in an early plan, the later plans' drift checks will (correctly) fire. Prefer keeping the documented contracts.
- Build test-first where the plan says "test-first" (002, 003, 004, 005, 006, 007). The RED step is required: see each failing test fail before writing the code that passes it.
